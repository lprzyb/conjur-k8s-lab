# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repo is

A single-VM lab that stands up CyberArk Conjur Enterprise alongside a standalone Kubernetes cluster, then demonstrates several ways an app (`cityapp`) can consume secrets from Conjur running as a follower in K8s: env-var hardcoding, push-to-file, push-to-K8s-secret, External Secrets Operator (ESO), and the Conjur CSI provider.

There is no build system, package manager, or test suite — this is a set of numbered shell scripts and Kubernetes YAML manifests meant to be run **in order, on a live Rocky Linux 9 VM acting as both the K8s node and the Conjur/podman host**. Nothing here runs or is testable from a local dev machine. (Originally written for CentOS Stream 9 — Rocky Linux 9 was chosen as the replacement because it's a strict 1:1 RHEL rebuild at the same EL9 major version, so no script changes were needed: `dnf`, SELinux, firewalld, and the EPEL9/`pkgs.k8s.io`/`isv:/cri-o` RPM repos this lab uses are all OS-agnostic at the EL9 level, not CentOS-specific.)

## Directory / execution order

Folders are numbered and must be run in sequence; scripts within each folder are also numbered and must be run in order:

1. `1.k8s-setup/` — installs CRI-O, kubelet/kubeadm/kubectl, creates the standalone K8s cluster, installs the K8s dashboard, and deploys a static landing page (`05.deploying-landing-page.sh`) that links to every demo's NodePort.
2. `2.conjur-setup/` — installs podman, runs MySQL (backing store for demo data), loads and starts the Conjur Leader container, loads demo policy/data, enables `authn-jwt-k8s`, deploys a Conjur Follower into K8s. See "Known architecture quirks" below for why the Leader GUI's `Settings > Conjur Cluster` page shows the Follower's status as "Unknown".
3. `3.cityapp-setup/` — builds the `cityapp` PHP image (`3.cityapp-setup/build/`) and deploys it in each secrets-integration mode (hardcode, conjurtok8sfile, conjurtok8ssecret, springboot-via-sidecar).
4. `4.cityapp-springboot/` — a Java/Spring Boot rewrite of cityapp (`build/` is a Maven project) demonstrating a *different* integration technique than folder 3: instead of an external sidecar/CSI injecting secrets with no app code changes, this app links CyberArk's `conjur-sdk-springboot` library and calls the Conjur API directly from `ConjurSpringDbConfig.java` to build its own DataSource at startup. Folder 3's `cityapp-springboot-sidecar.yaml` variant reuses this same built app but deployed via the sidecar pattern instead, for comparison. The two are deployed with distinct identities/ports (`cityapp-springboot-sidecar` on `30083`, `cityapp-springboot-native` on `30088`) specifically so both can run at the same time — they used to share a name/port and silently replace each other.
5. `5.conjur-eso/` — installs the External Secrets Operator, wires an ESO `SecretStore`/`ExternalSecret` to Conjur via JWT auth to sync `test/host2/*` into a native `conjur-secret` K8s Secret, then (`05.running-cityapp-eso.sh`) deploys the same `localhost/cityapp:php` image from folder 3, unmodified, mounting that Secret directly — demonstrating that once ESO has done the sync, the consuming app needs zero Conjur awareness (no ServiceAccount, no JWT token projection), unlike every sidecar-based variant in folder 3.
6. `6.conjur-csi/` — installs the Secrets Store CSI Driver and CyberArk's `conjur-k8s-csi-provider`, redeploys the Conjur Follower with both `authn-jwt/k8s` and a second, CSI-specific `authn-jwt/k8s-csi` authenticator enabled, then mounts `test/host1/*` into `cityapp-csi` via a `SecretProviderClass` — a third distinct integration technique alongside folders 3-5 (the CSI provider authenticates on the driver's behalf using an explicit `identity:` parameter, not JWT-claim auto-resolution, so the pod itself needs no `serviceAccountToken` projection).

Each numbered folder has a `yaml/` subfolder with the K8s manifests it applies, and every folder has a `00.config.sh` that must be edited and flipped to `READY=true` before other scripts in that folder will run.

## Config propagation pattern

- `1.k8s-setup/00.config.sh` is standalone (not sourced by anything downstream) and holds only `K8S_VERSION` — CRI-O and kubelet/kubeadm/kubectl are installed from version-matched repos (`isv:/cri-o:/stable:/$K8S_VERSION` and `pkgs.k8s.io/core:/stable:/$K8S_VERSION`), so this is the one value to bump when refreshing the K8s version; CRI-O and kubeadm must stay on the same minor version or the cluster won't come up.
- `2.conjur-setup/00.config.sh` is the root config for everything Conjur-related: VM/Conjur IP, domain, admin password, Conjur account, appliance tarball name/version, `CONJUR_LEADER_HOST` / `DB_HOST` (both derived from `LAB_DOMAIN`), demo DB credentials (`DB_ROOT_PASSWORD`/`DB_USER`/`DB_PASSWORD`), and `JWT_AUDIENCE`. `conjur_appliance_file` / `conjur_version` are only an *example* current version (last verified 13.7.0) — the appliance tarball is a licensed download from CyberArk (not fetchable via script), so whoever runs the lab must overwrite these with whatever they actually received from their CyberArk rep.
- `DB_HOST`/`DB_USER`/`DB_PASSWORD` and `JWT_AUDIENCE` each drive two independent things that must agree: the DB host/credentials are both the actual MySQL container (`02.running-mysql-db.sh`) and the value Conjur hands out (`07.loading-demo-data.sh`); the JWT audience is both what Conjur expects (`09.loading-conjur-jwt-data.sh`) and what the K8s follower requests in its token (`follower/follower.yaml`, templated in). Change these in `00.config.sh` only — never hardcode a new copy in a script.
- `5.conjur-eso/01.adding-conjur-eso-policy.sh` sets a *second* Conjur variable set, `test/host2/*` (distinct from `test/host1/*` used everywhere else), to the same `$DB_HOST`/`$DB_USER`/`$DB_PASSWORD` values — not placeholder text. It used to be seeded with fake illustrative values (`test2.demo.local` etc.) that didn't correspond to a real DB account; that was fine while the folder only proved "ESO can retrieve a value," but broke once `05.running-cityapp-eso.sh` started actually connecting to MySQL with them. If you ever want `test/host2/*` to demonstrate a distinct value again, keep in mind cityapp-eso needs it to be a working DB connection to show a real result. **`6.conjur-csi/` does not use `test/host2/*` at all** (its `SecretProviderClass` reads `test/host1/*`) — it used to also set `test/host2/*` to different, unrelated fake values, which silently clobbered folder 5's real ones if both folders were run in the same lab build. That was dead code and has been removed; don't reintroduce a write to `test/host2/*` from folder 6.
- This repo uses CyberArk's current terminology: **Leader** (not "master" — Conjur's own `evoke` CLI uses `evoke configure leader` / `--leader-altnames`, matching the "Follower" name already used everywhere) and, where relevant elsewhere, "primary" is not part of Conjur's vocabulary at all. Don't reintroduce "master" naming in new scripts/vars.
- `3.cityapp-setup/00.config.sh`, `4.cityapp-springboot/00.config.sh`, `5.conjur-eso/00.config.sh`, and `6.conjur-csi/00.config.sh` all `source ../2.conjur-setup/00.config.sh` then add their own `READY` gate — so editing `2.conjur-setup/00.config.sh` affects downstream folders too. `6.conjur-csi/00.config.sh` additionally defines `CSI_JWT_AUDIENCE` (default `conjur`), which is folder-local — it's not shared with the rest of the lab like `JWT_AUDIENCE` is, because the CSI provider's driver-level JWT request is a separate authentication flow from the app-level one.
- Every operational script (`0N.*.sh`) sources its folder's `00.config.sh` first and exits early if `READY` is not `true`:
  ```sh
  source 00.config.sh
  if [[ "$READY" != true ]]; then
      echo "Your configuration are not ready. Set READY=true in 00.config.sh when you are done"
      exit
  fi
  ```
  When adding a new script to a folder, follow this same guard pattern.
- Scripts wrap the substantive commands in `set -x` / `set +x` for transcript-style logging.

## Editing YAML manifests

- `2.conjur-setup/follower/follower.yaml` is a template: `11.deploying-follower-k8s.sh` copies it to `/tmp`, then `sed`-replaces the literal placeholders `CONJUR_IP`, `LAB_DOMAIN`, `CONJUR_VERSION`, `JWT_AUDIENCE` with the matching `00.config.sh` values before `kubectl apply`. If you add a new config-driven value to this file, add a placeholder + a corresponding `sed -i "s/PLACEHOLDER/$VALUE/g"` line, matching the existing ones.
- `3.cityapp-setup/yaml/*.yaml` manifests use a different (pre-existing) placeholder convention — curly braces, e.g. `{LAB_IP}`, `{LAB_DOMAIN}`, `{DB_HOST}`, `{JWT_AUDIENCE}`, `{DB_USER}`, `{DB_PASSWORD}` — substituted the same way by each `0N.running-cityapp-*.sh` script before `kubectl apply`. Keep new cityapp placeholders in this curly-brace style for consistency within the folder, even though it differs from folder 2's bare-token style. **Always quote a curly-brace placeholder that is the entire value** (e.g. `audience: '{JWT_AUDIENCE}'`, not `audience: {JWT_AUDIENCE}`) — YAML parses an unquoted value starting with `{` as an inline flow-mapping, not a string, so the source file is technically invalid YAML until `sed` has run. `mysql.{LAB_DOMAIN}`-style values are safe unquoted because the `{` isn't the first character.
- `5.conjur-eso/yaml/conjur-secret-store-jwt.yaml` uses the same curly-brace convention, templated by `02.creating-ext-secret-store.sh`: `{CONJUR_URL}`, `{CONJUR_ACCOUNT}`, `{JWT_AUDIENCE}`. `5.conjur-eso/yaml/cityapp-eso.yaml` uses `{LAB_IP}`/`{DB_HOST}`, templated by `05.running-cityapp-eso.sh`.
- `5.conjur-eso/yaml/conjur-external-secret.yaml`'s `secretKey` names (`dbaddr`/`dbuser`/`dbpass`) are deliberately chosen to match what `3.cityapp-setup/build/index.php`'s file-based lookup expects under `/etc/secret-volume` — that's what lets `cityapp-eso` reuse the cityapp image completely unmodified. If you change what `ExternalSecret` targets, keep those key names or update the app.
- `6.conjur-csi/yaml/04.conjur-csi-provider-class-config.yaml` uses `{CONJUR_URL}`/`{CONJUR_ACCOUNT}`, templated by `04.creating-secret-provider-class.sh`; `6.conjur-csi/yaml/05.cityapp-csi.yaml` uses `{LAB_IP}`/`{DB_HOST}`, templated by `05.running-cityapp-csi-test.sh`. `04.creating-secret-provider-class.sh` also appends the Conjur cert directly onto the end of the copied yaml file (indented, as a continuation of the `sslCertificate: |` block scalar) rather than templating it — **`sslCertificate: |` must stay the last line of that source yaml** or the appended cert lines will land in the wrong place.
- Everything else under each folder's `yaml/` and `policies/` is applied as static files — values like `demo.local`-shaped domains baked into other manifests, or the `jwt-apps/k8s` identity path used throughout `policies/authn-jwt-k8s.yaml`, are not templated and must be kept in sync with `00.config.sh` by hand. `jwt-apps/k8s` specifically is structural to the policy hierarchy (referenced many times as a path prefix within one file) rather than a deployment-specific value, so it's deliberately not parameterized.
- `1.k8s-setup/yaml/landing-page.html` is the one deliberate exception to the `sed`-templating convention: its demo links are rewritten **client-side**, by a small inline `<script>` that swaps each link's host for `location.hostname`. This was chosen over adding a `{LAB_IP}` placeholder because folder 1 runs before `2.conjur-setup/00.config.sh` (the source of `LAB_IP`) exists — sourcing it from folder 1 would mean either duplicating the IP in two config files or restructuring the whole `00.config.sh` sourcing chain (currently 3/4/5/6 → 2) to start one folder earlier. The JS is intentionally the only bit of client-side logic in this repo; keep it that way rather than reusing this pattern elsewhere.

## Known architecture quirks

- The Leader GUI's `Settings > Conjur Cluster` page shows the Follower's Services/Database/Free Space columns as "Unknown" — this is not a regression, the original repo's own reference screenshot (`images/08.conjur-follower.png`) shows the exact same gap. Root-caused via `podman logs conjur` and `kubectl -n conjur logs deployment/follower`, live, on a running lab:
  - **Confirmed real bug, fixed**: `2.conjur-setup/04.starting-conjur-container.sh` used to pass `--dns $CONJUR_IP` to `podman run` — pointing the Leader container's own DNS resolver at the VM's own IP, where nothing listens on port 53. Every DNS lookup the Leader's Ruby/UI backend made hung for the full UDP timeout before failing (`Nameserver ... not responding` / `No response from nameservers list`), making `Settings > Conjur Cluster` take 10+ seconds to load. This flag served no working purpose (nothing on the VM answers DNS at `$CONJUR_IP`) and has been present since the original repo's first commit of this file; it's been removed so the container falls back to podman's own working embedded resolver (page load dropped from ~10.5s to ~0.4s once fixed).
  - **Not the cause of the "Unknown" status, ruled out**: initially suspected the Leader does a live outbound poll to the Follower's `/health` endpoint and just can't resolve its K8s-internal hostname (`follower.conjur.svc.cluster.local`). A script (`12.linking-leader-to-follower-health.sh`, since removed) patched a matching `/etc/hosts` entry into the Leader container and confirmed `curl` to that hostname succeeds (`200`) from inside the Leader — but the Follower's status columns were still "Unknown" afterward, and `podman logs conjur`/`kubectl logs` captured across a live page refresh show **no outbound call from the Leader to the Follower at all, successful or failed**, and no heartbeat/report call from the Follower to the Leader either (only a one-time `Recording initial Leader ID: ...` at Follower pod startup). So the Leader isn't polling the Follower live for this data.
  - **Current best explanation, not yet independently confirmed**: `05.configuring-conjur-leader.sh` runs a plain `evoke configure leader` with no `--auto-failover`/etcd cluster setup — this lab is a simple Leader+Follower replication topology, not Conjur Enterprise's separate "Auto-Failover" clustering feature. The Leader's own row shows `Auto-failover: N/A` for the same reason. Working theory: the Leader's own Services/Database/Free-Space numbers are gathered locally (no network call needed to inspect itself), Replication status comes from Postgres's own `pg_stat_replication` view (confirmed real, matches the JSON `/health` endpoint's `database.replication_status` field), but the Follower's Services/Database/Free-Space columns specifically depend on the Auto-Failover cluster's etcd-based health-monitoring agent — never configured, not running, unrelated to DNS/networking. If accurate, "Unknown" is expected/correct for this lab's topology and not fixable without actually standing up Auto-Failover clustering (a much larger architecture change, out of scope unless a user asks for it).
  - **Also explained, cosmetic**: the "Domain Name"/"Host IP" columns are populated via reverse DNS on the connection source, not the node's own hostname. The Leader's own row shows a stray `,` and garbage like `b.ip6-servers.arpa` because one of its listed Host IPs is an IPv6 **link-local** address (`fe80::...%eth0`) — those aren't reverse-resolvable by any DNS server. The Follower's row shows `host.containers.internal` (podman's own bridge-gateway alias for `10.88.0.1`) instead of its real Pod IP, because the Follower reaches the Leader via the VM's own external IP (`CONJUR_IP`, same host, same VM), which loops back through podman's port-publishing NAT before the Leader ever sees the connection — functionally the same class of artifact the GUI's own disclaimer describes for a load balancer ("this may be the load balancer's IP/Domain Name, rather than the node's"), just caused by NAT hairpinning on a single-VM lab instead of a literal LB.
  - `04.starting-conjur-container.sh` also has a `conjur1.$LAB_DOMAIN` alias baked into its own `/etc/hosts` line (`grep -q "$CONJUR_LEADER_HOST" /etc/hosts || echo "$CONJUR_IP conjur1.$LAB_DOMAIN $CONJUR_LEADER_HOST" >> /etc/hosts`) that is dead: it's not part of `--leader-altnames` in `05.configuring-conjur-leader.sh`, not referenced anywhere else in the repo except a stale `README.md` line calling the container "conjur1" (it's actually named `conjur`, per `node_name` in `00.config.sh`). Flagged, not yet removed.

## cityapp UI conventions

- Deliberate constraint across this whole repo's front ends: **no JS frameworks, no Node.js/React, minimal CSS**. `3.cityapp-setup/build/index.php` uses plain PHP + hand-written inline `<style>` (no Bootstrap CDN — that was removed to also drop an external network dependency this lab doesn't otherwise need). It shows 5 random cities per load, a "last refreshed" timestamp, and a `<meta http-equiv="refresh" content="30">` tag so the page visibly updates on its own — matching the `conjur.org/secrets-refresh-interval: 30s` used throughout the annotation-based cityapp variants, and making Part IV's "change a secret and watch it update" story happen automatically instead of requiring a manual reload.
- The header logo/favicon used to be external `docs.cyberark.com`/`www.cyberark.com` URLs — both now 404 (the docs portal was restructured; confirmed live, deep links there rot easily and the site is a JS-rendered SPA with no discoverable static sitemap). Replaced with an inline CSS text wordmark and a data-URI SVG favicon so the page has zero external asset dependencies, consistent with the no-Bootstrap-CDN constraint above. If you're tempted to add a CyberArk doc link anywhere in this file, verify it resolves first (`curl -sL -o /dev/null -w '%{http_code}'`) — only `https://docs.cyberark.com/` (bare root) was confirmed working; every deeper path tried during this investigation 404'd.
- `index.php` reads a `DEMO_METHOD` env var (`hardcode` / `push-to-file` / `push-to-k8s-secret` / `eso` / `csi`, set one per Deployment: `3.cityapp-setup/yaml/cityapp-hardcode.yaml`, `cityapp-conjurtok8sfile.yaml`, `cityapp-conjurtok8ssecret.yaml`, `5.conjur-eso/yaml/cityapp-eso.yaml`, `6.conjur-csi/yaml/05.cityapp-csi.yaml`) to render a short "how this works" explanation card. This exists because the app's own credential-detection logic (`file_exists('/conjur/worlddb.json')` vs `/etc/secret-volume`) genuinely cannot distinguish `cityapp-conjurtok8ssecret`, `cityapp-eso`, and `cityapp-csi` from each other — all three mount to `/etc/secret-volume` and would otherwise show the identical "K8S SECRETS" label despite being three architecturally different mechanisms (sidecar push, external operator sync, CSI live-mount). `DEMO_METHOD` is presentation-only and deliberately doesn't touch the existing detection/connection logic. If you add a new cityapp variant, add a matching entry to `$demo_methods` in `index.php` and set `DEMO_METHOD` on its Deployment, or it'll just render without the explanation card (not an error, just missing the story).
- `1.k8s-setup/yaml/landing-page.html` is the one place with intentional client-side JS (see the note above under "Editing YAML manifests") — kept to a single small `forEach` loop, no library.

## Deliberately-pinned versions

- `1.k8s-setup/yaml/landing-page.yaml` uses `nginx:1.30-alpine` — nginx's "stable" branch (even minors), not `1.31.x`/"mainline" (odd minors) or the floating `:alpine` tag.
- `1.k8s-setup/yaml/kube-dashboard.yaml` stays on Kubernetes Dashboard **v2.7.0** (last version installable as a single raw manifest with NodePort access). v3+ requires Helm, cert-manager, and an ingress controller — out of scope for this lab's "one script, no extra dependencies" style. Don't bump this without also accepting that added complexity.
- Flannel is applied from `releases/latest/download/kube-flannel.yml`, not a branch URL, so re-running the script months later doesn't silently pull an unvetted manifest.
- `2.conjur-setup/06.installing-conjur-cli.sh` installs the CLI from `conjur-cli-go` (Go rewrite), not the old `conjur-api-python3` project — that project is deprecated (unmaintained since Sep 2024) and its download URLs may eventually 404. The Go CLI's flags for `init`/`login`/`policy load`/`variable set` are drop-in compatible with the old Python CLI (`-u`, `-a`, `-i`, `-f`, `-b`, `-v`, `-d` all still mean the same thing), except `conjur init` now requires an environment subcommand: use `conjur init self-hosted -u <url> -a <account> -s` for this appliance (self-hosted Conjur Enterprise, not `oss` or `saas`).
- `2.conjur-setup/follower/follower.yaml`'s `dap-seedfetcher` init container is pinned to tag `0.8` rather than the implicit `:latest`, for the same reproducibility reason as the Flannel pin.
- `2.conjur-setup/02.running-mysql-db.sh` uses `mysql:8.4` (current LTS) — MySQL 5.7 is EOL. The bundled `world.sql.gz` dump was generated by MySQL 8.0 and uses no version-specific syntax, so it loads cleanly on 8.4.
- `3.cityapp-setup/build/Dockerfile` is pinned to `php:8.4-apache` rather than the floating `php:apache` tag. `secrets-provider-for-k8s` is pinned to `:1.10.0` (was `:latest`) everywhere it's used in `3.cityapp-setup/yaml/`.
- The **secretless** cityapp variant (Secretless Broker sidecar) has been removed from this lab entirely — it's not just deprioritized, the files, README section, and its Conjur policy host entry are all deleted. Don't re-add it without the user asking.
- `3.cityapp-setup/yaml/cityapp-springboot-sidecar.yaml` and `4.cityapp-springboot/yaml/cityapp-springboot-native.yaml` both reference `localhost/cityapp-springboot:latest` — the *image* `4.cityapp-springboot/41.building-cityapp-image.sh` produces — instead of pulling the third-party `huydd79/cityapp-springboot:latest` (personal Docker Hub image, last pushed Dec 2022) directly. Note this is the image tag only; the k8s object names (`cityapp-springboot-sidecar` vs `cityapp-springboot-native`) are intentionally different from the image name and from each other. **Both cityapp-springboot deployments (folder 3's and folder 4's) now depend on `4.cityapp-springboot/41.building-cityapp-image.sh` having been run first** to populate `localhost/cityapp-springboot:latest` — there's no build step in folder 3 itself.
- The sidecar and native-SDK springboot variants used to share one ServiceAccount/Deployment/Service name and NodePort (`cityapp-springboot` / `30088`), so deploying one silently replaced the other. They're now fully separate: `cityapp-springboot-sidecar` (port `30083`, RBAC-listed in `3.cityapp-setup/yaml/conjurtok8ssecret-rbac.yaml`, Conjur host `system:serviceaccount:cityapp:cityapp-springboot-sidecar`) and `cityapp-springboot-native` (port `30088`, Conjur host `system:serviceaccount:cityapp:cityapp-springboot-native`, no RBAC needed since it never touches a k8s Secret) — both hosts are defined in `2.conjur-setup/policies/authn-jwt-k8s.yaml`. `cityapp-springboot-native.yaml` also had a leftover `conjur.org/container-mode: sidecar` annotation removed — it doesn't run a secrets-provider-for-k8s sidecar at all, so that annotation was never doing anything, just misleading.
- `41.building-cityapp-image.sh` tries the real local Maven/podman build first; if either step fails (`./mvnw clean package` or `podman build`) it falls back to pulling `huydd79/cityapp-springboot:latest` and retagging it as `cityapp-springboot` locally, so downstream yaml never needs to know which path was taken. This is a deliberate fallback, not a bug — keep it if you touch this script.
- `3.cityapp-setup/yaml/conjurtok8ssecret-rbac.yaml`'s Role is scoped to `resourceNames: ["db-creds"]` — it used to grant `get`/`update` on all Secrets in the `cityapp` namespace, which was needlessly broad for a demo whose whole point is showing a least-privilege secrets pattern. Keep any future Role here scoped to the specific Secret name.
- `4.cityapp-springboot/build/pom.xml`'s `com.cyberark:conjur-sdk-springboot` is pinned to `2.2.5` (was the Maven meta-version `LATEST`, which Maven has deprecated for years and doesn't reliably resolve outside a repository manager). `4.cityapp-springboot/build/Dockerfile` is pinned to `tomcat:9.0.119-jre17` (was `9.0.58-jdk17-openjdk-slim` — ~60 patches behind; note Tomcat's official image dropped the `-openjdk-slim` tag suffix entirely in favor of `-jreNN`/`-jdkNN`, so the old tag family is no longer being published, only still pullable because Docker Hub doesn't delete old tags). `application.properties`'s `spring.datasource.driver-class-name` was fixed from the legacy `com.mysql.jdbc.Driver` to the current `com.mysql.cj.jdbc.Driver` to match the `com.mysql:mysql-connector-j` dependency actually declared in `pom.xml`. The Dockerfile's `ADD target/*.war /usr/local/tomcat/webapps/` used to deploy the WAR under its own filename (`pom.xml`'s `<finalName>cityapp</finalName>` → `cityapp.war`), which Tomcat auto-maps to context path `/cityapp/`, not root — both `cityapp-springboot-sidecar` (`30083`) and `cityapp-springboot-native` (`30088`) returned Tomcat's own 404 page at the root URLs the README documents. Fixed by removing Tomcat's default `ROOT` webapp and deploying the WAR as `ROOT.war` instead, so it takes over the root context like every other cityapp variant in this lab.
- **Known, unresolved tech debt**: `4.cityapp-springboot/build/pom.xml` is on `spring-boot-starter-parent` **2.7.14** — Spring Boot 2.x reached OSS end-of-life 2023-06-30 (only paid Tanzu support remains, through end of 2026). `conjur-sdk-springboot` itself supports Spring Boot 3.x, but upgrading this app is a real code migration, not a version bump: `City.java` uses `javax.persistence.*` (→ `jakarta.persistence.*` under Boot 3), the JSTL dependency and JSP taglibs are javax-based, and the Tomcat base image would need to move to a Jakarta EE 9+ (Tomcat 10+) build. Not attempted — no local Java/Maven toolchain was available to verify a migration compiles, and the blast radius (multiple source files) is bigger than everything else fixed in this pass. Revisit deliberately with the user before touching it.
- `5.conjur-eso/00.installing-eso-helm.sh` (renamed from a typo'd `00.installing-eso-heml.sh`) now pins `helm install external-secrets ... --version 2.7.0` instead of installing whatever's latest at run time. `5.conjur-eso/yaml/conjur-secret-store-jwt.yaml` and `conjur-external-secret.yaml` were bumped from `apiVersion: external-secrets.io/v1beta1` to `external-secrets.io/v1` — ESO dropped v1beta1 support in v0.17.0 (chart 2.7.0 requires it). `5.conjur-eso/02.creating-ext-secret-store.sh` also switched its `sed` delimiter from `/` to `#` so `CONJUR_FOLLOWER_URL` no longer needs its slashes pre-escaped — a fragile pattern that would silently break if the URL were ever edited without also updating the escaping.
- `6.conjur-csi/00.installing-csi-helm.sh` (renamed from `00.installing-csi-heml.sh`) pins the `secrets-store-csi-driver` chart to `--version 1.6.0`; `6.conjur-csi/03.installing-conjur-csi-provider.sh` pins `conjur-k8s-csi-provider` to `--version 0.2.5`. Neither had a version pin before.
- `6.conjur-csi/yaml/04.conjur-csi-provider-class-config.yaml`'s `conjur.org/configurationVersion` was bumped `0.1.0` → `0.2.0`. This is a real format change, not just a number: 0.2.0+ ignores the `secrets` parameter on the `SecretProviderClass` entirely and expects a `conjur.org/secrets` annotation on the consuming pod instead — `6.conjur-csi/yaml/05.cityapp-csi.yaml`'s Deployment template now carries that annotation (moved verbatim from the old `SecretProviderClass`). 0.1.0 was still functionally supported when this was written, so this wasn't required to keep the demo working — it's a deliberate upgrade to the currently-documented pattern, done per CyberArk's `conjur-k8s-csi-provider` README, not live-tested against a running cluster.
- `6.conjur-csi/yaml/02.follower-with-csi.yaml` (a separate copy of the follower manifest — it needs `CONJUR_AUTHENTICATORS: authn-jwt/k8s,authn-jwt/k8s-csi` instead of just `authn-jwt/k8s`) had the same unpinned `dap-seedfetcher` and hardcoded `audience: cybrdemo` as folder 2's copy did before that was fixed — pinned to `:0.8` and templated to `JWT_AUDIENCE` the same way.
- `6.conjur-csi/yaml/01.conjur-csi-jwt-policy.yaml`'s `k8s-csi-host` had its `authn-jwt/k8s-csi/kubernetes.io/serviceaccount/name: k8s-csi` restriction commented out, leaving only the namespace check — restored it (narrowing-only, and `05.cityapp-csi.yaml` already creates and uses a ServiceAccount named exactly `k8s-csi`, so there's no reason it wouldn't match). Also added an `audience` variable to this policy, set from `$CSI_JWT_AUDIENCE` in `01.adding-conjur-csi-jwt-policy.sh` — previously the CSI driver's Helm install requested tokens with a specific audience (`--set 'tokenRequests[0].audience=conjur'`) but Conjur was never told to validate it, so that claim went unchecked.
- Deliberately **not** set for `conjur/authn-jwt/k8s-csi`: `token-app-property` / `identity-path`. Those exist on the main `authn-jwt/k8s` authenticator to let Conjur auto-resolve a host identity from JWT claims, but the CSI provider passes an explicit `identity: host/jwt-apps/k8s-csi/k8s-csi-host` parameter in the `SecretProviderClass` instead — adding auto-resolution config here would be redundant, not a fix for a gap.

## Checking for drift: check-versions.sh

`./check-versions.sh` (repo root) checks every pinned version listed under "Deliberately-pinned versions" above against its live upstream release (K8s, dap-seedfetcher, MySQL's LTS line, secrets-provider-for-k8s, PHP, conjur-sdk-springboot via Maven Central, Tomcat, the external-secrets Helm chart). It's a maintenance tool, not part of the numbered lab pipeline — no `00.config.sh`, no `READY` gate, never edits a file, and is safe to run from anywhere with `curl`/`jq` (including the lab VM itself; per-check network failures are reported as `MANUAL`, not fatal).

- **Intentional caps are marked `PINNED`, not `STALE`** — e.g. Kubernetes Dashboard (capped at v2) and Spring Boot (EOL, migration deferred) always report this way regardless of what upstream does next; don't "fix" the script to make these show `OK`.
- The MySQL check resolves Docker Hub's `lts` alias to whichever numeric tag currently shares its digest, rather than comparing against the highest `X.Y` tag — Docker Hub also publishes a separate, faster-moving "Innovation" line (`9.x` at the time this was written) under plain version tags, which is not what `2.conjur-setup/02.running-mysql-db.sh` should track.
- When adding a new pinned dependency anywhere in the lab, add a matching check here (extract the current value straight from its source file, don't hardcode it in the script) so this tool stays a single source of truth instead of drifting out of sync with the pins themselves.
- Update the maven artifact/Docker Hub/GitHub identifiers here if CyberArk ever renames these projects again (as happened with the CLI → `conjur-cli-go` and the "Idira"/product-name changes noted elsewhere in this file).
